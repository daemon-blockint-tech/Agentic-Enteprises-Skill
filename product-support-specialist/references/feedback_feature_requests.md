# Feedback and Feature Requests

## Feedback vs bug vs support

| Type | Definition | Destination |
|------|------------|-------------|
| **Bug** | Broken vs spec | Engineering via `support-engineer` |
| **Feedback** | Improvement to working product | Product backlog |
| **Question** | Unclear how to use | KB / training |

## Capture template

```markdown
**Title:** [verb + object + outcome]
**Product area:** …
**Persona:** role / plan
**Problem:** When I …, I can't …
**Impact:** time, revenue, risk (customer words)
**Frequency:** one-off / recurring in tickets
**Workaround:** yes/no
**Customer willing to interview:** Y/N
**Ticket IDs:** …
```

## Prioritization signals (for PM, not support to decide)

| Signal | Weight |
|--------|--------|
| ARR of accounts asking | High |
| Blocker vs nice-to-have | High |
| Aligns with stated strategy | Medium |
| Low effort workaround exists | Lowers urgency |

Support **does not commit** roadmap dates.

## Customer-facing language

**Good:**
> I've logged this as feedback for our product team. We can't share a timeline, but your use case is noted under [ID].

**Avoid:**
> We'll build this next quarter.

## Deduping

- Search existing feedback IDs before creating
- Link related tickets to one master request
- Note segment patterns (enterprise vs SMB)

## Closing the loop

When shipped (from release notes):

- Proactive email to requesters when policy allows
- Update KB if feature changes workflow

## Product gap themes → KB

If same feedback weekly:

- Draft KB "limitations and alternatives"
- Suggest macro for agents
- Escalate to PM for positioning or build

## Integration with monetization

Upgrade requests masquerading as feedback → clarify plan fence with `product-management-monetization`; log if fence is unclear.
